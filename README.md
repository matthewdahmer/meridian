# Meridian

Chandra X-ray Observatory spacecraft thermal visualization suite. Meridian is a static multi-page website composed of three tools:

- **Pitch Sensitivity (Protractor)** — shows the maximum allowed dwell time per thermal model as a function of pitch angle, under coupled thermal constraints. Rendered as a semicircular polar plot with color-coded constraint bands.
- **Steady States** — shows Xija model steady-state temperatures as multi-line dwell curves and summary tables, one model at a time, across pitch angles and mission dates.
- **Model Dashboard** — shows historical model performance for each Chandra thermal model: telemetry vs. time, residual scatter, error histogram, pitch-bin error traces, and a configurable dwell exploration scatter plot.

All three apps share the same dark sidebar theme, Bootstrap 5, and D3 v7. All vendor assets are served locally — no CDN dependencies.

## Development Server

No build step is required for development. Serve the repository root with Python's built-in HTTP server:

```bash
cd meridian/
python -m http.server 8765
```

Then open:

| App | URL |
|-----|-----|
| Pitch Sensitivity | http://localhost:8765/protractor/ |
| Steady States | http://localhost:8765/steady_states/ |
| Model Dashboard | http://localhost:8765/model_dash/ |

> **Hard refresh:** `Cmd+Shift+R` (macOS) after editing `app.js` to bypass the browser cache.

## Generating Steady States Data

The Steady States app requires data files generated by the `xija_steady_states` package, which lives as a sibling directory at `../xija_steady_states/`. It is not assumed to be installed — run it directly from its source tree. This step requires the `ska3-dev` conda environment and access to the Chandra engineering archive.

```bash
conda activate ska3-dev
cd ../xija_steady_states/
SKA=~/AXAFDATA/SKADATA python -m xija_steady_states --output-dir ../meridian/steady_states
```

This generates `meridian/steady_states/data/` containing:
- `manifest.json` — list of all 16 thermal models
- `<nav_id>/metadata.json` — per-model config and file index
- `<nav_id>/<nav_id>_<config_id>_<date>.json` — per-date chart data (28 dates per config)
- `<nav_id>/<nav_id>_<config_id>_<date>.csv` — per-date dwell CSV
- `<nav_id>/<nav_id>_<config_id>_table.csv` — summary table across all dates

Generation covers all 16 models and all 40 configurations (roughly 30–60 minutes). To restrict to specific models or dates:

```bash
SKA=~/AXAFDATA/SKADATA python -m xija_steady_states \
    --output-dir ../meridian/steady_states \
    --models aca dpa fptemp \
    --dates 2026001 2026182
```

To regenerate without skipping existing files:

```bash
SKA=~/AXAFDATA/SKADATA python -m xija_steady_states \
    --output-dir ../meridian/steady_states \
    --force
```

The protractor's `data/` directory is managed separately and is not generated by this command.

## Generating Model Dashboard Data

The Model Dashboard requires one gzip-compressed JSON file per model in `model_dash/data/`, plus a `manifest.json`. These are generated by a separate data pipeline (not part of this repository). The data files are not automatically created by any command in `meridian/` — they must be placed in `model_dash/data/` before the dashboard will show any content.

The `manifest.json` format:

```json
[
  { "nav_id": "1dpamzt", "display_name": "ACIS Electronics (DPA)" },
  { "nav_id": "4rt700t", "display_name": "OBA Forward Bulkhead" }
]
```

Each `nav_id` must match the `msid` field inside the corresponding `<nav_id>.json.gz` file exactly (case-sensitive), because `nav_id` is used for the unit lookup in `MSID_DISPLAY_UNITS`.

The full schema for `.json.gz` files is documented in `model_dash/data_file_structure.md`.

## Building for Deployment

`build.py` assembles all three apps into a single `dist/` directory ready to copy to a web server.

```bash
cd meridian/
python build.py
```

This copies `protractor/`, `data/`, `steady_states/`, and `model_dash/` (including generated data) into `dist/`. Run with `--force` to overwrite an existing `dist/`:

```bash
python build.py --force
```

To build into a different directory:

```bash
python build.py --output-dir /path/to/webroot --force
```

Verify the result by serving it locally:

```bash
python -m http.server 8765 --directory dist
```

> `build.py` warns but does not fail if `steady_states/data/` or `model_dash/data/` is absent — generate the data first before a production build.

### Protractor (manual)

The protractor app has no automated test suite. Verify manually after changes:

1. Start the dev server (`python -m http.server 8765` from `meridian/`)
2. Open `http://localhost:8765/protractor/`
3. Confirm the protractor plot renders with colored bands
4. Change the Date and Chips dropdowns — plot and tables should update
5. Change a variable limit dropdown — plot re-renders without a network fetch
6. Expand the "Steady States" sidebar section — model links should appear (requires `steady_states/data/` to exist)
7. Click a model link — navigates to `steady_states/?model=<nav_id>` and pre-selects that model

### Model Dashboard (manual)

The Model Dashboard has no automated test suite. Verify manually after changes:

1. Start the dev server (`python -m http.server 8765` from `meridian/`)
2. Open `http://localhost:8765/model_dash/`
3. Confirm the sidebar populates with model names (requires `model_dash/data/manifest.json`)
4. Select a model — provenance card, quad plot, error segments card, and dwell exploration card should all render
5. Test box zoom (drag) and double-click reset in each chart
6. Test axis linking: zooming the time series X axis should update the error-time chart; zooming the scatter Y should update the time series Y
7. Click the legend in the telemetry vs. time plot — it should cycle through four corners
8. Change X/Y axis dropdowns in the Dwell Exploration card — plot and title should update
9. Toggle the "Jitter scatter" checkbox — scatter dots should jitter horizontally
10. Select a second model — all three cards should be destroyed and re-rendered cleanly
11. Verify the URL updates (`?model=<nav_id>`) on each model selection, and that loading the URL directly pre-selects the correct model

## Navigation

The sidebar is shared across all three apps:

- **Pitch Sensitivity** — links to `protractor/`
- **Steady States** — 16 model links load content in-place (SPA, no page reload)
- **Model Dashboard** — model links load content in-place; URL updates to `?model=<nav_id>` on each selection

Clicking a model in the Steady States sidebar from within the protractor navigates to `steady_states/?model=<nav_id>`, which auto-selects that model on load. The Model Dashboard sidebar is populated from `model_dash/data/manifest.json` at page load.

## Data Notes

- `data/` (protractor) and `steady_states/data/` (steady states) and `model_dash/data/` (model dashboard) contain large binary/JSON files. Do not read individual scenario files into context — use the schema descriptions in `CLAUDE.md` and `model_dash/data_file_structure.md` instead. The `manifest.json` in each directory is small and may be read freely.
- Protractor scenario files may contain literal `NaN` (not valid JSON). The app replaces `NaN` with `null` before parsing.
- Steady states data files are standard JSON with no `NaN` values.
- Model dashboard data files are gzip-compressed JSON (`.json.gz`). The app decompresses them in the browser using the `DecompressionStream` API. All temperature and residual values in the files are always in °C regardless of the model's display unit; the app converts to °F at render time for applicable models.
